## 1. 目的

本文書は、要求定義を完成させるために確定した判断と、実装・data・外部承認として残るgapを分離して管理する。文書へgapを書いたことはgapの解消を意味しない。

status:

| Status               | 意味                                      |
| -------------------- | ----------------------------------------- |
| `open`               | owner判断、実装、data修正、検証が必要     |
| `externally_blocked` | 外部承認・証跡なしには解消できない        |
| `decided`            | 方針は確定したが、実装・検証は別途必要    |
| `resolved`           | 完了条件とverification evidenceがそろった |
| `not_applicable`     | 適用しない決定と根拠が記録済み            |

critical／high gapはproduction release前に0件でなければならない。medium以下もowner、review target、acceptanceまたは明示的なrisk acceptanceを必要とする。各本文表のOwnerを省略した場合も次のregistryを継承し、`unassigned`を黙認しない。

全Owner roleは個人開発者1名が兼務する。role名は増員や相互承認を前提にせず、責務と証跡の分類labelとして使用する。

## 7. Gap closure procedure

gapを`resolved`にするcommit／change recordは次を含む。

1. GAP IDとaffected requirement／spec／AC
2. before evidenceとroot cause
3. decision、implementationまたはdata correction
4. regression fixture
5. exact verification command／manual procedureとresult
6. traceability matrix、acceptance status、必要なrunbook／schema更新
7. ownerとreview date

critical／high gapのseverity低下やrisk acceptanceは、単独開発者の暗黙判断にせずProduct Ownerの明示記録を必要とする。

## 8. Documentation-baseline status

本文書作成時点で、`tags.zip`の全entry、既存spec、確認できたcurrent implementationとの差、external policy requirementは文書上の反映先を持つ。新しい差分を発見した場合は本台帳へ追加し、網羅性の主張を固定化しない。残る項目は上記のproduct／data／external gapsであり、実装済みとして隠していない。
