## 1. サービス概要

`diopside` は、白雪巴の公開YouTubeアーカイブを、あとから探しやすく、見返しやすく、内容を把握しやすくするためのアーカイブ検索・分析サービスである。

YouTube上の動画メタデータ、配信予定、公開チャット、リプレイチャットを収集し、動画一覧、構造化タグ、検索インデックス、チャット集計、ワードクラウド、タイムスタンプ候補などの公開データへ変換する。利用者は静的UIから、スマートフォン中心にアーカイブを検索・閲覧できる。diopside独自タグはYouTubeの公式カテゴリと区別し、大分類・小分類・不変IDを持つ検索facetとして提供する。

運用面では、個人開発で継続できることを重視し、常時起動サーバーやRDB、OpenSearchなどの固定費が大きい構成を避ける。収集・処理・公開のtarget architectureはDynamoDB、S3、SQS/Lambda、CloudFrontを候補とするが、現行リポジトリの実装範囲はNuxtフロントエンドと開発用seedである。targetとcurrentの境界は [00.index.md](00.index.md) を正とする。

## 2. 解決したい課題

### 2.1 視聴者側の課題

YouTubeアーカイブは本数が増えるほど、目的の配信を探しにくくなる。タイトルや投稿日だけでは「どんな内容だったか」「どの配信で話していたか」「盛り上がった場面がどこか」を思い出しにくい。

特に、次のような探し方はYouTube単体では難しい。

- 「歌枠だけをまとめて見たい」
- 「以前話題になった雑談回をもう一度見たい」
- 「チャットが盛り上がっていた場面を見たい」
- 「長時間配信の中で見どころから入りたい」
- 「初見でも入りやすいアーカイブを探したい」
- 「スマートフォンで片手操作しながら軽く探したい」

### 2.2 二次創作・切り抜き作業側の課題

切り抜きやまとめ作業では、長時間アーカイブから見どころを探す必要がある。手作業で配信全体を確認するには時間がかかる。チャットの反応、頻出語、盛り上がり区間、概要欄タイムスタンプなどを横断して確認できると、作業の初動が大きく短縮される。

### 2.3 運用者側の課題

個人開発では、便利な検索・分析機能を作れても、運用費と保守負荷が高くなると継続できない。YouTube API quota、Lambda実行時間、DynamoDB item size、S3保存量、CloudFront cache、DLQ、再実行、障害調査を低コストで管理できる設計が必要である。

### 2.4 タグ分類・監査側の課題

平坦なタグ名だけでは、ゲームジャンルの「ホラー」と中心テーマの「ホラー」、歌種別の「ライブ」と同時視聴メディアの「ライブ」を区別できない。旧名、表記ゆれ、複合タグ、出演者とグループ、作品名と一般ジャンルが混在すると検索結果の意味が壊れる。

新着動画を継続して追加するには、全動画を同じtaxonomyで処理し、各assignmentの理由と根拠を追跡し、未知語や競合を公開前にreviewできる必要がある。件数と必須fieldがそろう構造監査だけでなく、動画内容と分類ruleが一致するsemantic auditも必要である。

## 3. diopsideが提供する価値

### 3.1 探せる価値

動画タイトル、タグ、投稿日、長さ、配信種別、チャット集計、生成成果物の有無を使って、目的のアーカイブを素早く見つけられる。

代表例:

- 歌枠、雑談、ゲーム、ASMRなどのタグ検索
- キーワード検索
- 年別・月別・日付別の探索
- ランダム発見
- 最近検索・閲覧履歴からの再訪

### 3.2 内容をつかめる価値

動画を開く前に、チャット集計、頻出語、ワードクラウド、タイムスタンプ候補を確認できる。長時間アーカイブでも、全体の雰囲気や見どころを短時間で把握できる。

代表例:

- チャット件数、投稿者数、スーパーチャット系件数
- 頻出語ランキング
- タイムライン別チャット量
- ワードクラウド画像
- 概要欄・チャット盛り上がり・キーワードスパイク由来のタイムスタンプ候補

### 3.3 見返しやすい価値

気に入った動画や後で見たい動画を保存し、閲覧履歴から再訪できる。モバイルファーストのUIにより、スマートフォンで隙間時間に探し、気になったアーカイブをYouTubeで開ける。

### 3.4 作業を助ける価値

切り抜き、まとめ、ファン活動、アーカイブ整理の初動を支援する。人が最終判断する前提で、候補地点や候補語を自動生成する。

### 3.5 運用を継続できる価値

固定費が大きい構成を避け、S3静的配信、DynamoDB single-table、Lambda/SQSバッチ、EventBridge低頻度実行を中心にすることで、個人開発でも長く運用できる。

### 3.6 分類を信頼できる価値

タグを大分類・小分類・不変IDで管理し、aliasをcanonical tagへ統合する。利用者は同名異軸を混同せず、作品、出演者、グループ、企画、テーマ、公開形式を組み合わせて探せる。運用者は付与理由、source、evidence、confidenceから誤分類を再確認し、再生成後の差分を監査できる。

## 9. プロダクト原則

1. まず探しやすくする。
2. 長時間アーカイブの理解コストを下げる。
3. 生成結果は候補として提示し、最終判断は人に委ねる。
4. 公開情報だけを扱う。
5. チャット本文はS3、DynamoDBはmanifest/summaryに限定する。
6. APIより静的公開データを優先し、閲覧コストを下げる。
7. 失敗してもJobEvent、DLQ、debug artifactで追跡できるようにする。
8. モバイルファーストで、説明なしに使えるUIにする。
9. 個人開発で継続できるコストに抑える。
10. [00.index.md](00.index.md) の正本順位に従い、要求、詳細仕様、AC、実装差分を双方向traceabilityで管理する。
11. 平坦な表示名をtag identityにせず、分類軸とstable tagIdを保持する。
12. 構造監査だけで意味的正確性を主張せず、根拠とsemantic fixtureを確認する。
13. 配信本文の言及を実出演と分離し、根拠不足の候補を公開しない。
14. partial metadataの不明値を0や汎用labelで偽装せず、取得範囲と精度を表示する。

## 11. まとめ

`diopside` は、単なる動画一覧サイトではなく、公開アーカイブを「探す」「理解する」「見返す」「運用する」ための支援サービスである。

視聴者には、構造化tagと静的検索から本人アーカイブと精査済み外部コラボを探し、本文で触れた人物・別コンテンツを手がかりに再発見し、動画内容を把握しやすい体験を提供する。切り抜き・まとめ作成者には、長時間配信から候補区間を見つけるための手がかりを提供する。タグ監査者には、分類理由とsourceを追跡し、意味的誤分類や外部候補の誤採用を公開前に修正できるworkflowを提供する。運用者には、低コストで収集・集計・公開を継続し、失敗時に追跡・再実行できる基盤を提供する。

最終的な価値は、増え続ける公開アーカイブを、時間が経っても見つけられ、理解でき、楽しみ直せる状態に保つことである。
