## 1. 目的

本文書は、動画タグの分類、付与、正規化、根拠、版管理、本文言及、外部コラボ発見、公開投影、検索、監査、移行を定義する。タグは検索用の平坦な文字列集ではなく、「何をしているか」「何の作品か」「誰が参加するか」「どの企画か」「どのような文脈か」「本編で何へ言及したか」「どう公開されたか」を独立した軸で表現する。

対象:

- metadataに存在する全対象動画
- 保存Wiki HTMLで確認した外部配信と、共演者名・group aliasから解決したチャンネルの全公開動画走査・精査結果
- taxonomy、一般別名、複合旧タグ、グループ別名
- 動画への根拠付きtag assignment
- 字幕・保存コメントからの言及候補、採否、時刻付き根拠、検出coverage
- 外部コラボの候補、除外理由、未解決チャンネル、取得不能metadataを含む発見ledger
- 公開tag registry、tag index、動画一覧への投影
- 新規動画追加時の生成、review、承認、監査
- v1の整数tag index／平坦名から構造化tagへの移行
- タグfacet、URL共有、検索、解除、0件回復の利用者体験

対象外:

- タグ補正専用の管理画面
- 未承認語からのcanonical tag自動新設
- YouTube source tagsをそのままdiopside分類タグとして公開すること
- コメント本文や投稿者識別子をタグ根拠として公開すること

管理画面は対象外でも、運用者が候補をreviewし、承認後に再生成・監査できるworkflowは対象内とする。

## 2. 用語

| 用語                 | 定義                                                                              |
| -------------------- | --------------------------------------------------------------------------------- |
| source tag           | YouTube metadataの`snippet.tags`。分類判断の入力であり、diopside tagではない      |
| canonical tag        | 正規化・alias解決後のタグ定義                                                     |
| tag assignment       | 一つの動画と一つのcanonical tagの根拠付き関連                                     |
| `tagId`              | 表示名変更後も変わらないタグ定義の不変ID                                          |
| taxonomy             | 大分類、小分類、固定値、基数、適用条件の版付き定義                                |
| review state         | 判定不能、競合、未知語をタグ配列へ保存せず保留する作業状態                        |
| public projection    | audit向けassignmentから公開検索に必要な最小項目だけを生成した成果物               |
| legacy composite key | `(categoryId, subcategoryId, canonicalName)`。`tagId`移行前だけ使用する暫定識別子 |
| structural audit     | field、型、件数、基数、参照、重複を機械的に検証する監査                           |
| semantic audit       | 動画固有の内容と付与理由を人または根拠付き判定で検証する監査                      |

`confidence` はタグ判断の確からしさであり、workflow状態ではない。`high` と `medium` だけを確定assignmentに許可し、未確認はreview stateに置く。

### 5.3 共通分類規則

- `content.primary` と `format.media` は各1件だけとする。
- `people.channel` はmetadataから1件を導出する。
- 複合内容は中心目的を主ジャンルとし、副ジャンルを最大2件付ける。
- 誕生日、周年、大会、コラボ、お披露目は開催理由・状況であり、実際の中心内容を主ジャンルにする。
- `Shorts`、公式切り抜き、3D、縦型、MVは公開・制作形式であり主ジャンルにしない。
- `その他`、`不明`、`要確認`を確定tagとして保存しない。
- `オフコラボ`には`コラボ`も付ける。
- ゲームジャンル`ホラー`を付けた同じ意味で中心テーマ`ホラー`を重複付与しない。
- `マシュマロ`は投稿媒体であり雑談種別にしない。
- グループは構成員一致だけで自動確定せず、グループとしての明示または公式seriesを根拠にする。
- 本文で名前へ触れただけの人物を`people.performer`へ入れず、`reference.mentionedPerson`へ分離する。同一人物が実出演した場合はperformerを優先し、両方へ重複させない。
- 本文言及の根拠不足はreview stateとし、確定assignmentやpublic projectionに含めない。「検出できない」と「言及がない」を同義にしない。
- people分類を除くtag数12件はhard maximumではなくreview thresholdとする。超過だけでassignmentを削除せず、重複軸、episode marker、generic作品名、根拠の弱い副tagを人が確認する。

### 5.4 主ジャンル境界

| 主ジャンル | 含める                                 | 除外する                                        |
| ---------- | -------------------------------------- | ----------------------------------------------- |
| ゲーム     | gameplay、実況、ゲーム由来の切り抜き   | ゲームの感想だけを話す雑談                      |
| 雑談       | 会話、相談、体験談、振り返り、投稿紹介 | 会話量が多くてもgameplayが中心の動画            |
| 歌         | 歌唱、歌枠、cover、original、音楽live  | 雑談中に数曲だけ歌う動画                        |
| ASMR       | ささやき、耳かき、添い寝等の音響体験   | 通常雑談で小声になっただけの動画                |
| 企画       | 検証、quiz、講座、review、お披露目等   | ゲーム大会、歌企画など明確な内容genreがある動画 |
| 同時視聴   | 特定作品を視聴しながら反応する内容     | 視聴後の感想・振り返りだけの動画                |
| TRPG       | session進行・roleplay本編              | session後の振り返り雑談                         |
| 朗読・声劇 | 台本・物語の朗読、掛け合い、演技       | ゲーム内textを読む実況                          |
| ダンス     | 踊ってみた、dance中心の動画            | 歌liveの一部だけのdance                         |

## 6. 正規化、alias、グループ

### 6.1 照合正規化

照合keyには次を順に適用する。表示名は公式表記を保持する。

1. 前後空白を除去する。
2. Unicode NFKCを適用する。
3. 連続空白をASCII空白1個へ置換する。
4. 英字の大文字・小文字を区別しない。
5. 照合時だけ先頭の`#`を除去する。

人名から敬称を除き、作品名・series名から回数、話数、`#1`、`最終回`を除く。年ごとに区別すべき大会・eventは西暦4桁を残す。絵文字をtag名に含めない。

## 7. 判定根拠とreview

### 7.1 根拠優先順位

1. 動画タイトル
2. 概要欄の動画固有部分
3. 公式hashtag、series表記、出演者表記
4. 字幕、チャット、コメント

概要欄の定型channel説明、販売link、SNS、credit、共通hashtagを根拠から除外する。字幕からtheme tagを追加する場合は、タイトル・概要欄で予告されているか、連続10分以上または本編15%以上で中心的に扱われることを確認する。チャット・コメントだけで内容tagを確定しない。

### 7.2 自動判定と人手確認

| 根拠                                        | 扱い                               |
| ------------------------------------------- | ---------------------------------- |
| canonical名または既知aliasがtitleに完全一致 | 自動確定候補                       |
| 概要欄の動画固有部分に明示                  | 自動確定候補。出演者は誤検出を確認 |
| title構造と既存seriesから高確度推定         | 自動付与後に差分review             |
| 字幕・chatの意味推定                        | 候補だけ生成し人手確認             |
| 未知語、`その他`、主ジャンル競合            | 必ず人手確認                       |

未知語から新規canonical tagを自動生成してはならない。追加時は包含基準、除外基準、alias、影響件数、migration、versionを同時に承認する。

### 7.3 配信本文の言及判定

- WebVTT字幕を主根拠とし、保存コメントは時刻付き要約または複数の独立コメントが同じ発言を示す場合に使う。
- title・概要欄ですでに扱っている対象を「本文から新発見した言及」としない。
- 短い人名、一般語と重なる人名、ゲーム内キャラクターと同名の人物は、敬称、周辺文脈、反復、別根拠を組み合わせる。
- 十分な根拠のある候補だけを`accepted`とし、弱い一致は`reviewCandidate`として確定assignmentから分離する。手動採用・却下はactor、reason、source record、時刻、rules versionを持つ。
- 字幕がない動画もコメント根拠が十分なら採用できるが、取得対象外を「言及なし」と表示しない。

### 7.4 外部コラボの発見と精査

1. 保存Wiki HTMLの各年「ライブ配信（アーカイブ）」節から、本人チャンネルを除くYouTube videoIdを抽出する。削除・非公開・API取得不能もledgerから除外しない。
2. 根拠付きassignmentと動画固有概要欄から共演者のcanonical集合を作り、YouTubeチャンネル解決の成否と解決根拠を保存する。未解決者を欠落扱いせず明示する。
3. 解決済みチャンネルのuploads全件についてtitleと動画固有概要欄を走査し、出演、通話、同時プレイ、歌唱・音声・映像提供、企画回答、共同制作の候補を作る。
4. 白雪巴を構成員に持つgroupについて、本人以外の解決済み実メンバー側uploadsをgroup正規名・aliasで別経路走査する。
5. 名前やgroupを話題にしただけ、単独の事前練習・振り返り、商品告知、過去動画link、大会名簿、定型creditは除外し、除外理由と証拠snippetを保持する。
6. Wiki、共演者名精査採用、group alias精査採用の和集合を統合台帳とし、出典の複数性を失わない。未確定`review`は統合台帳へ入れない。

## 10. 公開データ投影

### 10.1 Pathと原子的切替

versioned releaseの正本:

```text
/data/latest.json
/data/releases/{releaseId}/index.json
/data/releases/{releaseId}/search-index.json
/data/releases/{releaseId}/tag-taxonomy.json
/data/releases/{releaseId}/tag-index.json
/data/releases/{releaseId}/tag-alias-index.json
/data/releases/{releaseId}/videos/{videoId}.json
```

現行フロント互換alias:

```text
/data/index.json
/data/videos/{videoId}.json
/data/wordcloud/{videoId}.svg
```

互換aliasはcanonicalとschemaが異なるためredirectにせず、検証済みreleaseからcurrent型の内容を原子的にmaterializeする。`latest.json`と各aliasが異なるreleaseを示す期間を5分以内にし、部分releaseを参照してはならない。

### 10.2 最低schema

```ts
type PublicTagEntry = {
  readonly tagId: string
  readonly categoryId: string
  readonly category: string
  readonly subcategoryId: string
  readonly subcategory: string
  readonly displayName: string
  readonly count: number
  readonly videoIds: readonly string[]
}

type PublicVideoBase = {
  readonly videoId: string
  readonly title: string
  readonly sourceKind: 'owned_channel' | 'external_collaboration'
  readonly tagIds: readonly string[]
  readonly artifactFlags: {
    readonly chat: boolean
    readonly comments: boolean
    readonly timestamps: boolean
    readonly wordcloudChat: boolean
    readonly wordcloudComments: boolean
    readonly wordcloudBoth: boolean
  }
  readonly chatCount?: number
}

type PublicVideoSummary = PublicVideoBase &
  (
    | {
        readonly metadataStatus: 'resolved'
        readonly publishedAt: string
        readonly publishedAtPrecision: 'instant'
        readonly durationSec: number
        readonly channelTitle: string
        readonly thumbnail: {
          readonly url: string
          readonly width: number
          readonly height: number
        }
        readonly sourceUpdatedAt: string
      }
    | {
        readonly metadataStatus: 'wiki_only'
        readonly publishedDate: string
        readonly publishedAtPrecision: 'day'
        readonly wikiPage: string
        readonly wikiHostLabel?: string
      }
  )
```

上記のnormal用tag schemaを含むcanonical envelope、purge時のforbidden fields、hash／path、detail／alias型は[22.system-specifications.md](22.system-specifications.md) SPEC-DATA-PUB-001を正とする。公開`index.json`は`releaseMode=normal`時に`schemaVersion`、`releaseId`、`generatedAt`、`taxonomyVersion`、`aliasVersion`、`tagIndexPath`、`videos`を持ち、tag entryは`tag-index.json`だけを正本とする。`PublicTagEntry.videoIds`は昇順stable sequenceとし、`count`はそのunique件数と一致する。`search-index.json`は`schemaVersion`、`releaseId`、`releaseMode`、`generatedAt`、`normalizationVersion`、`videos`を持ち、normalの各videoは`videoId`、`titleTokens`、`sourceKind`、`metadataStatus`、`tagIds`、日付値と精度、`artifactFlags`を持つ。`durationSec`で絞り込む場合、`wiki_only`は値不明として範囲一致に含めず、longest sortでは既知durationの後へ配置する。reason／source／evidence、字幕・コメントsnippetを含めない。

### 10.3 公開rules

- `tag-index` はtagIdからvideoIdを逆引きできる。
- 同じdisplay nameでもtagIdまたは分類軸が異なれば別entryとする。
- URL queryの正本はtagIdとする。
- alias入力はcanonical tagIdへ解決し、共有URLをcanonical表現へ置換する。
- category、subcategory、displayNameは表示用でありidentityに使わない。
- taxonomy、alias、tag index、video indexは同じreleaseIdを参照する。
- reason、source、evidence、sourceTags、review stateを公開しない。
- `wiki_only`は日付精度、metadata不足、host identity未確定をUIで区別できる最小fieldだけを投影し、取得済みmetadataと同じ表示にしない。
- discoveryの候補snippet、概要欄全文、reviewスコア、除外理由、未解決チャンネル調査記録はprivate auditに保持し、public projectionへ含めない。
- 動画cardの表示tagは公開チャンネルを除き、主ジャンルを優先して最大2件とする。検索条件自体は表示上限に左右されない。

### 10.4 `tagId`移行

1. ingest時だけlegacy composite keyを使用する。
2. migrationは各keyへ不変tagIdを一度割り当て、version管理registryへ保存する。
3. 割当後のtagIdをcanonicalNameから再計算しない。
4. export前に全assignmentをtagIdへ変換する。
5. canonical versioned JSONは`tagIds`だけを持ち、legacy `tags`を併記しない。初回targetで現行3 pathのproduction projectionを生成しない。
6. migration inputのflat／comma入りdisplayNameはoffline mappingでcanonical tagIdへ解決し、同名異軸と3 comma fixturesを文字列分割で壊さない。0または複数候補はreviewへ送る。
7. contract testはpublic JSONにflat `tags`が0件で、全tagIdsがregistryと双方向一致することを検証する。
8. production URLはrepeated `tag=<tagId>`だけを受理し、display-name／comma区切り`tags` keyをtag条件へ適用しない。

## 14. 運用と変更

1. 本人metadata、保存Wiki HTML、共演者・group registryをversionでfreezeする。
2. Wiki台帳更新、共演者チャンネル解決・uploads走査、group alias走査、各候補精査を行い、統合台帳を固定する。
3. taxonomy、alias、group、spoken-reference rules、registry、algorithmをversionで固定する。
4. 本文言及と本人・外部tag候補を生成し、review対象を公開集合から分離する。
5. semantic fixture、候補差分sample、partial metadata fixtureを人が確認する。
6. AC-TAG-01、02、07、08、09を実行する。
7. JSON合格後にExcelとpublic projectionを生成する。
8. AC-TAG-03、04、05、06を実行する。
9. 全gate合格後にreleaseを切り替え、変更理由、影響件数、version、承認者、rollback対象を保存する。

受領資料が参照する`output/`、`external_output/`、`html/`、Wiki更新・共演者走査・group走査・精査・統合・metadata補完・本文言及・tag生成・Excel生成・監査の各scriptはこのリポジトリにもZIPにもない。復元されるまで再生成手順を実行可能と記載しない。

## 15. 完了条件

tag領域は次をすべて満たした場合だけ適合済みと判定する。

- FR-TAG、DR-TAG、NFR-TAGの全Must要件に実装と証拠がある。
- AC-TAG-01〜09が合格する。
- 本人チャンネルと外部コラボのapproved populationが`scopeDecisionVersion`で固定され、未解決・取得不能・除外を含むdiscovery ledgerが全件追跡可能である。
- review state、unknown tagId、semantic blocking defectがproduction集合にない。
- target public schemaと現行フロントのmigrationが完了し、flat表示名identityを廃止できる。
- Policy gateが有効で、独自tagのprovenanceを明示する。
- taxonomy、alias、registry、assignment、Excel、public projection、文書が同じreleaseへ追跡できる。
- 実施していない生成、監査、UI、accessibility検証を合格と記録していない。
