## 2. 判定規則

### 2.1 状態

| 状態           | 使用条件                                                               |
| ---------------- | ---------------------------------------------------------------------- |
| `not_run`        | fixture／environmentは定義済みだが、assertionを実行していない          |
| `blocked`        | 実装、data、credentialを含まない外部証跡、owner decisionがなく実行不能 |
| `pass`           | 全独立assertionが同一target versionに対して合格しevidenceが保存済み    |
| `fail`           | 1件以上のassertionが不合格                                             |
| `not_applicable` | 条件付き要件を採用しない決定、理由、approverが記録済み                 |

AC全体は全Then assertionのANDとする。skip、未測定、`unknown`をpassへ数えない。再実行時はtarget commit／release、fixture version、command、environment、開始／終了時刻、各assertion結果を更新する。

### 2.2 必須メタデータ

各実行recordは次を持つ。

- AC ID、assertion ID、normal／error／boundary／security／accessibility／recovery等のType
- 関連要件、SPEC ID、GAP ID
- target commitまたはreleaseId、schema／algorithm／taxonomy version
- fixture ID／hash、environment、browser／viewportまたはservice emulator
- exact commandまたはmanual steps、exit status
- 観測値、期待値、結果
- 証拠path／dashboard／承認済み記録、実行者、reviewer、時刻

external policy evidenceにsecretやpersonal dataを含めず、scope、project、feature、retention、effective／expiry dateを監査可能にする。

### 2.3 受け入れ属性の既定値

registry／本文で省略した属性は次を継承する。

| 属性            | Default／override                                                                                                                                                        |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 種別                 | 3節registryまたは各AC本文の種別                                                                                                                                          |
| Confidence           | confirmed。owner承認が必要なcandidate thresholdを判定するAC-FE-01／02、AC-OPS-02／03、AC-COST-01、AC-QUOTA-01、AC-DATA-02、AC-NFR-01／03はQ-008 closureまでinferred      |

## 4. 受け入れケース

本節を全AC本文の唯一の正本とする。`20` v1.2.0まで同文書に置かれていた既存AC本文を意味変更なしで移設し、追加ACと一体で管理する。

各受け入れ条件のThenに列挙した期待結果は、それぞれ独立したassertionとして合否を記録する。一つでも不合格なら、その受け入れ条件全体を不合格とする。

### AC-FE-01: 最新アーカイブへ到達する

- Given: 公開indexに動画が20件以上あり、開発者以外1名以上が初回利用する同意記録のない375px幅のbrowserと、quick tag／newest／random／keyword／年月から異なる期待videoへ到達する代表5 taskがある
- When: 各taskで規約へ同意し、候補を選び、動画詳細からYouTube導線を開き、browser戻る操作でホームまたは検索へ戻る
- Then:
  - AC-FE-01-A: 60秒以内に正しいvideoIdのYouTube URLへ到達する
  - AC-FE-01-B: 戻る操作後に同じホーム一覧が表示される
  - AC-FE-01-C: 戻る操作後にホーム一覧のスクロール位置が保持される
  - AC-FE-01-D: page headingとservice purpose、quick tags、newest、randomが仕様順に表示される
  - AC-FE-01-E: quick tagsが0件でもrandom discovery入口が表示され、対象集合にあるvideoIdだけを開く
  - AC-FE-01-F: newestは`publishedAt`desc／`videoId`stable順で、cardにtitle、date、duration、thumbnail、優先tag最大2件、存在時だけchat countがある
  - AC-FE-01-G: cardの明示detail actionとYouTube actionが同じ正しいvideoIdへ遷移する
  - AC-FE-01-H: 代表5 taskのうち4件以上で、同意画面表示から期待videoのYouTube URL到達まで60秒以内である

### AC-FE-02: 複合条件で検索する

- Given: 有効な同意記録、各queryの期待video集合を持つ20件の固定検索評価データ、タグ、年月、動画長、チャット有無、sort同値fixture、NFR-PERF-001の代表的ミドルレンジ端末がある
- When: 20 queryで語句と3種類以上の条件および各sortを設定し、結果一覧から詳細へ遷移してbrowser戻る操作を行う
- Then:
  - AC-FE-02-A: 全条件を満たす動画だけが表示される
  - AC-FE-02-B: 条件変更から100ms以内に結果が更新される
  - AC-FE-02-C: URL再読込後も同じ条件と結果になる
  - AC-FE-02-D: 0件時に解除可能な条件候補が1件以上表示される
  - AC-FE-02-E: 詳細から戻った後に検索条件と検索結果が保持される
  - AC-FE-02-F: 20 query中16件以上で期待videoが先頭10件以内に含まれる
  - AC-FE-02-G: newest／oldest／longest／mostChatの全結果がSPEC-UI-SEARCH-001の比較keyとvideoId tie-breakerどおりである
  - AC-FE-02-H: URLのNFC keywordとNFKC／case-fold／空白正規化後の同値keywordが同じtitleTokens集合と結果へ解決する

### AC-FE-03: 配信内容を把握する

- Given: metadata、chat、comments、wordcloud、timestampsがある動画と、一部がない動画がある
- When: 各動画詳細を開く
- Then:
  - AC-FE-03-A: 存在する成果物だけが表示される
  - AC-FE-03-B: 各成果物にsource、coverage、生成時刻が表示される
  - AC-FE-03-C: 欠落値は0として表示されない
  - AC-FE-03-D: タイムスタンプ候補から同じatSecのYouTube再生位置を開ける

### AC-FE-04: 保存・履歴を管理する

- Given: localStorageが利用可能である
- When: 動画を保存し、詳細を閲覧し、canonical検索条件をrecent searchに保存し、再読込後に保存／履歴／recent searchを開き、1件ずつ個別削除してから全削除する
- Then:
  - AC-FE-04-A: 再読込後に保存した動画が再表示される
  - AC-FE-04-B: 再読込後に閲覧履歴が新しい順で再表示される
  - AC-FE-04-C: 全削除後は対象localStorage keyに利用者データが残らない
  - AC-FE-04-D: recent searchはkeywordだけでなくcanonical tagId、length、date、artifact、sortを含む条件全体を復元する

## 8. Release受け入れ規則

`releaseMode=normal`のproduction releaseは次のANDで受け入れる。

1. GATE-001〜006がvalid
2. affected Must／Should requirementのACがpassまたは承認済み変更
3. critical／high gapが0件
4. schema、reference、private-field、security、tag structural／semantic checksがpass
5. relevant performance、a11y、browser、cost、quota、recovery evidenceがcurrent release／configに対応
6. exact commands、manual checks、not-run itemsがreview可能

GATE待ちでも実装と固定fixture検証が完了した状態は`release candidate`と記録できるが、product DoD、production acceptance、公開完了とは扱わない。初回production normal releaseはGATE-001〜006の全合格を必須とする。metadata-only表示状態はrelease modeではなく、stagingではnormal相当buildの検証構成、受入後の失効containmentでは`compliance_purge`で実現し、Must／Shouldやproduct acceptanceを合格にしない。

`releaseMode=compliance_purge`は上記のproduct acceptanceの例外であり、GATE無効化・削除・retention反映のためだけにpublishしてよい。AC-DATA-01-A〜P、AC-FE-07、AC-DATA-02が同じpurge releaseと対象削除に合格し、新規データ・派生データ・tagを追加しない場合だけ安全化切替として受け入れる。特にMのmanifest join、NのreleaseId一貫性／対象URL遮断、Oのcontainment profile、Pのcache contractを省略しない。これをGATE-001〜006、Must／Should requirement、product releaseの合格に数えない。

文書baselineはproduct releaseではない。current frontendを配信中であっても、target acceptanceが未達ならtarget statusを`implemented`または`complete`にしない。
