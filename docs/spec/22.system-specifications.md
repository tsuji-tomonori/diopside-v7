## 1. 目的と原則

本文書は、要求を実装・test可能な画面、data、API、job、運用contractへ落とす。framework、cloud SDK、IaC tool等の交換可能な実装選択は固定せず、外部から観測できる入力、出力、状態、error、security boundaryを固定する。

共通rule:

- current implementationとtarget contractを明記し、未実装targetをcurrentと呼ばない。
- 欠落値を0、空配列、成功状態で偽装しない。
- private／audit／public dataをfield単位で分離する。
- state transition、retry、timeout、idempotency、versionを明示する。
- 外部PolicyとGATEは機能より優先する。
- UI detailは`docs/design/design-system.md`のtoken／component ruleに従う。

### SPEC-UI-SEARCH-001: Search、suggest、filter、sortの仕様

- 要件: BR-001、BR-009、BR-011、FR-FE-002、FR-FE-003、FR-TAG-011、014、022、NFR-PERF-002
- 種別: screen／interaction
- 対象: `/search`
- 確度: confirmed
- 出典: UC-02、SRC-DESIGN、SRC-TAG、現行検索実装
- 受け入れ: AC-FE-02、AC-FE-08、AC-TAG-03、AC-TAG-04、AC-TAG-09、AC-NFR-01

条件:

| 条件         | 正規値 representation            | Rule                                                                                    |
| ----------------- | ----------------------------------- | --------------------------------------------------------------------------------------- |
| keyword           | singleton `q=<UTF-8 RFC3986 value>` | Unicode NFC、trim、連続空白1個、最大200 code points。termsのAND                         |
| tag               | repeated `tag=<tagId>`              | 1〜20件、duplicate除去、tagId辞書順、選択tagIdsのAND、同名異軸を区別                    |
| length min/max    | singleton `lmin` / `lmax`           | 0以上の10進整数分、inclusive、`durationSec >= lmin*60`かつ`<= lmax*60`、5h+は`lmax`省略 |
| date from/to      | singleton `from` / `to`             | `YYYY-MM-DD`のUTC calendar date、inclusive、`from <= to`                                |
| artifact presence | repeated `artifact=<enum>`          | enumは`chat`、`comments`、`wordcloud`、`timestamps`、AND、表の順、GATE失効時は除外      |
| sort              | singleton `sort=<enum>`             | `newest`、`oldest`、`longest`、`mostChat`。`mostChat`はGATE-001合格時だけ               |

canonical query keyの順序は`q`、全`tag`、`lmin`、`lmax`、`from`、`to`、全`artifact`、`sort`とし、`sort=newest`も省略しない。singleton keyの重複は先頭のvalid valueだけを採用しvalidation noticeを表示する。unknown key／value、空value、無効date／numberを条件に適用せず、取得・解析の前に上記順のcanonical URLへ`replace`する。正規化後`q`が200 code points超、unique valid `tag`が20件超、または`lmin > lmax`のrequestは値を黙ってtruncate／swapせず、該当条件を不適用にしてvalidation noticeと修正actionを表示する。URL、router state、検索実行条件は同じ判定結果を使う。

keywordはURL保存時のNFCに加え、照合時にmanifestの`normalizationVersion`で次の同一tokenizerをtitleとqueryへ適用し、全query tokenが`titleTokens`に存在する動画だけを候補にする。(1) Unicode 15.1でNFKC、full case fold、(2) Unicode White_SpaceをASCII space 1個へ畳みtrim、(3) Letter／Mark／Number以外をtoken境界とする、(4) Han／Hiragana／Katakanaを含むrunはextended grapheme clusterのunigramと隣接bigram、その他のrunはrun全体をtokenとする、(5) 空tokenを除き最初の出現順で重複除去する。`normalizationVersion`はUnicode version、上記rule version、実装artifact SHA-256を解決でき、version不明なら検索artifactをrejectする。ASCII、全半角、濁点、emoji、結合文字、日本語1文字／2文字／無空白文のknown-answer fixtureをproducerとbrowserで共有する。sort比較keyは次を正本とし、全比較の最後に`videoId asc`を必須tie-breakerとする。

| sort       | Comparison key                                                                 |
| ---------- | ------------------------------------------------------------------------------ |
| `newest`   | `publishedAt desc`, `videoId asc`                                              |
| `oldest`   | `publishedAt asc`, `videoId asc`                                               |
| `longest`  | `durationSec desc`, `videoId asc`                                              |
| `mostChat` | `chatCount` present first, `chatCount desc`, `publishedAt desc`, `videoId asc` |

`metadataStatus=wiki_only`のdate-only recordは`publishedDate`をcalendar dateとしてnewest/oldestへ含めるが、同日の正確な時刻順を捏造せず、同日内はmetadata取得済みrecordの時刻順の後に`videoId asc`で並べる。duration不明recordはlength filterへ一致させず、`longest`では既知durationの後に置く。`mostChat`で`chatCount`欠落videoは全てのpresent videoより後ろに置き、いずれの欠落値も0へ変換しない。

初回targetのURLはrepeated `tag=<stable-tagId>`だけを受理する。現行のdisplay name／comma区切りqueryをproduction互換として維持せず、unknown／非canonical tag値は適用しない。frontend内部の保存条件は同じcanonical queryへ一度に移行する。

suggestは最大4 rows（tag最大2、date最大1、keyword最大1）。tag rowはsubcategory context、name、countを持つ。keyboardのArrow／Enter／Escで操作し、screen readerへname、role、positionを通知する。filter UIはmobile bottom sheet、desktop 320px right panelとし、live result count、reset、apply／closeを提供する。

- 要件: BR-009、FR-TAG-011、014、NFR-TAG-007、CON-008
- 種別: component／interaction
- 対象: SearchBar、SuggestList、ConditionRow、ConditionSheet／Panel、VideoListItem／VideoGridCard
- 確度: confirmed

### SPEC-UI-STATE-001: Loading、empty、partial、error、recoveryの仕様

- 要件: FR-FE-006、FR-WC-004、NFR-A11Y-003
- 種別: cross-screen state
- 対象: all public screens
- 確度: confirmed
- 出典: error／empty requirements
- 受け入れ: AC-FE-07、AC-FE-10

| State                | 判定                                | 表示／action                                 |
| -------------------- | ----------------------------------- | -------------------------------------------- |
| loading              | request idle/pending                | skeletonまたはstatus text。emptyを表示しない |
| empty                | request success、valid data count 0 | reasonと条件解除／homeへのaction             |
| partial              | index成功、optional artifact欠落    | 存在項目だけ表示し、欠落を0で偽装しない      |
| recoverable error    | timeout、5xx、一時network           | reason category、retry                       |
| permanent/not found  | 404、unknown videoId                | not-found、search／home                      |
| known video fallback | detail artifact失敗、videoId valid  | YouTube direct action                        |

state changeはfocusを奪わず、screen readerへ簡潔に通知する。retryはduplicate navigation／duplicate saveを起こさない。

## 3. データ仕様

### SPEC-DATA-META-001: Video metadataと状態

- 要件: FR-META-001〜006、DR-001、CON-002
- 種別: data
- 対象: raw／control／public
- 確度: confirmed
- 出典: SRC-YT-META、SRC-YT-POLICY、SRC-IMPL
- 受け入れ: AC-ING-01、AC-ING-02、AC-COMP-02

target channelはversion付き設定`channelIds`で1件以上指定し、初期値は白雪巴の1 channelだけとする。設定変更はscope decision、GATE-002、対象集合diffを再検証する。

video identityは11文字のYouTube `videoId`とし、titleやURLをkeyにしない。control recordは最低限次を持つ。

| 項目                                                    | 種別／constraint                                   | Null rule                             |
| -------------------------------------------------------- | -------------------------------------------------- | ------------------------------------- |
| `videoId`、`channelId`                                   | non-empty string                                   | null禁止                              |
| `title`、`description`                                   | string                                             | API欠落時だけnull可                   |
| `publishedAt`                                            | RFC 3339 instant                                   | API欠落時はnull                       |
| `scheduledStartTime`、`actualStartTime`、`actualEndTime` | RFC 3339 instant                                   | 未確定／非liveはnull                  |
| `duration`、`durationSec`                                | source ISO 8601／non-negative integer              | 未取得時はnull。0で代用しない         |
| `liveBroadcastContent`、`liveState`                      | source value／upcoming・live・archive・unavailable | source欠落はnull／derived stateは必須 |
| `privacyStatus`、`embeddable`                            | source value                                       | 未取得時はnull                        |
| `thumbnail`                                              | source URL、width、height                          | source欠落時はnull                    |
| `sourceTags`、`youtubeCategoryId`                        | source metadata                                    | diopside tagと分離                    |
| `viewCount`、`likeCount`、`commentCount`                 | non-negative integer                               | 非返却値はnull                        |
| `etag`、`fetchedAt`                                      | string／RFC 3339                                   | null禁止                              |

state transitionは`upcoming -> live -> archive`を通常経路とし、任意stateから`unavailable`へ遷移できる。`unavailable`から復帰した場合は新しい`fetchedAt`と根拠を記録する。同一`etag`の再取得はsource freshnessだけを更新し、派生jobを重複起動しない。

### SPEC-DATA-CHAT-001: Chat、comment、replay inputの仕様

- 要件: FR-CHAT-001〜008、FR-COM-001〜006、DR-008
- 種別: data
- 対象: raw／processed
- 確度: confirmed
- 出典: SRC-YT-LIVE、SRC-YT-COMMENT、SRC-UC-INPUT
- 受け入れ: AC-ING-03〜05、AC-COMP-03

rawはsource payload、request metadata、取得時刻、content hashをprivateに保持する。replay importは`videoId`、`source`、`acquiredAt`、`acquisitionMethod`、`permissionEvidenceRef`、`contentHash`を持つmanifestと不可分とする。証跡欠落、未知video、schema不正、重複hashは`quarantined`とし、processedへ渡さない。

normalized eventのidentityは`sourceType + sourceId`とし、更新型eventは`updatedAt`とcontent hashで上書きする。`messageId`／`commentId`の重複は件数を増やさない。unknown eventはraw参照、type、発生時刻を保持して隔離し、pipeline全体を停止しない。

chat common eventのdiscriminatorは`text_message`、`super_chat`、`super_sticker`、`member_milestone`、`membership_gifting`、`poll`、`deletion_tombstone`、`ban`、`chat_ended`、`unknown`とする。comment raw recordは`commentId`、`parentId`、`videoId`、`textOriginal`、`authorChannelId`、`authorDisplayName`、`publishedAt`、`updatedAt`、`likeCount`、`visibilityState=public_at_fetch`、`fetchedAt`を持つ。APIが返さないparent／author／updatedAt／likeCountはnullまたは省略し、0／空文字へ変換しない。公開APIが返さないpending／held／spam等のmoderation stateを生成しない。

coverageは最低限`coverageStart`、`coverageEnd`、`completeFromStart`、`sourceUpdatedAt`を持つ。取得開始前のchatを未取得の場合、`completeFromStart=false`とし、UI・public dataで「全件」と表現しない。

### SPEC-DATA-NORMALIZED-001: 正規化eventと集計

- 要件: FR-AN-001〜007、DR-005、DR-007
- 種別: data
- 対象: processed
- 確度: confirmed
- 出典: UC-09、privacy requirements
- 受け入れ: AC-AN-01、AC-AN-03、AC-SEC-02

normalized eventは`sourceType`、`sourceId`、`videoId`、`eventAt`、`relativeSec`、`text`、`eventType`、`paidKind`、`amountMicros`、`currency`、`authorDedupToken`、`schemaVersion`を持つ。sourceTypeはchat／comment、sourceId／videoId／eventAt／eventType／schemaVersionは必須である。`relativeSec`は動画基準時刻との差で0以上、基準不明またはcommentならnullとする。textのないsystem eventだけtext=null、paidKind／amountMicros／currencyはpaid eventだけ必須で他eventは禁止する。authorDedupTokenはsource authorがあり保持許可中だけprivateに存在し、publicへ出さない。

text normalizationはversioned configにUnicode data version、NFKC、full case fold、URL pattern、control／newline rule、言語別tokenizer、stopword set hash、min grapheme count、emoji policy、implementation hashを固定する。同じconfigとinputのknown-answer token列を2回一致させ、unknown config versionをrejectする。

public aggregateは個別本文、display name、author ID、支払額を含めず、chat／commentを別seriesとして保持する。combined viewを作る場合は両series、coverage、対象期間を併記する。`uniqueAuthorsApprox`はvideo-scoped HMACだけから計算し、video間照合へ使わない。

### SPEC-DATA-ART-001: Wordcloudとtimestamp候補

- 要件: FR-WC-001〜005、FR-AN-005、FR-AN-006、NFR-PERF-004
- 種別: artifact
- 対象: processed／public
- 確度: confirmed
- 出典: UC-03、UC-04
- 受け入れ: AC-AN-02、AC-AN-03、AC-NFR-01

wordcloud JSONはvideoId、sourceSet、status、coverage、tokenizerVersion、stopwordVersion、algorithmVersion、generatedAtを持つ。`status=generated`の場合だけ安全化済みSVG、`topTerms`、`svgPath`、`svgSha256`を持つ。語数不足時は`status=not_generated`、`notGeneratedReason=insufficient_terms`とし、SVGを生成せずpath／hashを出力しない。

private `WordFilterConfig`はversion、NG語set hash、PII-like pattern version、URL／連続数字rule、単一author低頻度threshold、effectiveAtを持つ。変更履歴はbefore／after config hash、actor、changedAt、reasonをappend-onlyで記録する。filter後のNG語、PII canary、URL、連続数字、単一authorだけの低頻度語をpublic topTerms／SVGへ出さない。

timestamp candidateは`candidateId`、`videoId`、`atSec`、`label`、`evidenceType`、`confidence`、`coverage`、`algorithmVersion`を持つ。`atSec`は0以上かつ動画長以内とし、概要欄、chat量、語句spikeのどれを根拠としたか識別する。人の確定chapterとは別typeにする。

### SPEC-DATA-PUB-001: 正規の原子的公開release

- 要件: BR-011、CON-003、FR-EXP-001〜005、FR-FE-002、004、010、FR-TAG-022、DR-TAG-016、NFR-PERF-002、004、NFR-REL-001、003、NFR-SEC-002、NFR-TAG-012
- 種別: data／publish contract
- 対象: target static export and frontend
- 確度: confirmed
- 出典: SRC-IMPL、SRC-TAG、SRC-YT-POLICY
- 受け入れ: AC-FE-05、AC-FE-07、AC-DATA-01、AC-EXP-01、AC-COMP-03、AC-NFR-01、AC-TAG-03、AC-TAG-09

初回targetは次のcanonical pathだけをproduction contractとする。frontendとexportを同じtarget releaseで移行し、現行の`/data/index.json`、`/data/videos/{videoId}.json`、`/data/wordcloud/{videoId}.svg`を互換aliasとして維持しない。外部consumerが確認された場合だけ、利用実態と期限を根拠に変更管理で互換要件を追加する。

```text
/data/latest.json
/data/releases/{releaseId}/index.json
/data/releases/{releaseId}/search-index.json
/data/releases/{releaseId}/tag-taxonomy.json
/data/releases/{releaseId}/tag-index.json
/data/releases/{releaseId}/tag-alias-index.json
/data/releases/{releaseId}/videos/{videoId}.json
/data/releases/{releaseId}/wordcloud/{videoId}-{source}.svg
/data/releases/{releaseId}/wordcloud/{videoId}-{source}.json
```

全JSON objectは`additionalProperties: false`、未定義の`null`は禁止、optional fieldは省略する。`releaseId`は`^[A-Za-z0-9][A-Za-z0-9._-]{0,127}$`、`videoId`は`^[A-Za-z0-9_-]{11}$`、hashはlowercase SHA-256 hex 64文字、時刻はUTC RFC 3339、count／durationは0以上の安全整数とする。pathは同じrelease rootの正規化relative pathだけを許し、absolute、`.`／`..`、backslash、query、fragment、control character、二重encodingを拒否する。

最低schemaは次と同値とする。

```ts
type ReleaseMode = 'normal' | 'compliance_purge'

type ArtifactEnvelope = {
  readonly schemaVersion: string
  readonly releaseId: string
  readonly releaseMode: ReleaseMode
  readonly generatedAt: string
}

type LatestManifest = ArtifactEnvelope & {
  readonly normalizationVersion: string
  readonly indexPath: string
  readonly searchIndexPath: string
  readonly artifactHashes: Readonly<Record<string, string>>
  readonly tagTaxonomyPath?: string
  readonly tagIndexPath?: string
  readonly tagAliasIndexPath?: string
  readonly purgeBaseReleaseId?: string
  readonly purgeBaseManifestSha256?: string
  readonly purgeTrigger?: string
}
```

### SPEC-DATA-JOB-001: Job、checkpoint、監査event

- 要件: FR-OPS-001〜008、NFR-REL-002
- 種別: control data
- 対象: private control plane
- 確度: confirmed
- 出典: UC-06〜UC-10
- 受け入れ: AC-OPS-01〜05

`JobEvent`は全stateで`jobId`、`canonicalJobKey`、`parentJobId`（rootのみ省略）、`jobType`、`targetId`、`inputVersion`、`status`、`attempt`、`queuedAt`、`inputManifest`、`correlationId`、`quotaUnitsByMethod`を持つ。`startedAt`、`endedAt`、`outputManifest`、`errorCode`、retry／DLQ fieldはstatus discriminatorどおりだけにrequired／forbiddenとする。`canonicalJobKey`は`jobType`、`targetId`、`inputVersion`をこの順にserializeする。各値をraw UTF-8 bytesへ変換し、4-byte unsigned big-endian byte lengthと値bytesを連結した全体のSHA-256 lowercase hexを使う。文字数ではなくbyte数を長さとし、区切り文字・Unicode再正規化を加えず、caller headerから作らない。ASCII、multi-byte、空文字、境界値のknown-answer fixtureで全producer／consumerの同値性をcontract testする。

状態は`queued -> running -> succeeded | failed_retryable | failed_permanent | cancelled`とし、`failed_retryable(retryDisposition=scheduled) -> queued`だけを自動遷移に使う。`attempt`は初回実行を1とし、自動retry ordinal 1〜3がattempt 2〜4である。retryable errorのattempt 1〜3はscheduled、attempt 4は`exhausted_to_dlq`としてDLQへ移す。delayはretry ordinal `r`に対して`full-jitter uniform [0, min(8, 2 * 2^(r-1))] seconds`とし、外部`Retry-After`が1〜3,600秒ならその値を下限にして上限3,600秒へclampする。fake clockと注入済み乱数0／1境界で2／4／8秒のbase、8秒cap、3 retry、attempt番号を検証する。`failed_permanent`は自動retry 0回で初回失敗からDLQへ移す。cancelは`queued`またはsafe checkpointを持つ`running`だけで受理し、`succeeded`を取り消さない。redriveは元job、修正理由、actor、時刻を新jobへ関連付ける。

### SPEC-OPS-PUBLISH-001: 検証、publish、rollback

- 要件: BR-005、FR-EXP-001〜005、NFR-REL-003
- 種別: release flow
- 対象: target pipeline
- 確度: confirmed
- 出典: SRC-UC-REPO
- 受け入れ: AC-EXP-01、AC-DATA-01、AC-OPS-02

release validation stateは`building -> validating -> ready | rejected`とし、`validating -> rejected`だけがvalidation failure、両terminal stateからの再利用は禁止する。publicationは別のappend-only pointer event（`publish`、`supersede`、`rollback`）で管理し、ready releaseだけをpublishできる。後続publishは旧active releaseへsupersede eventを記録し、rollbackはpolicy-compliantな既存ready／過去published releaseを参照する新pointer eventとして記録する。release object自体のvalidation stateを巻き戻さない。`releaseMode=normal`のreadyはschema、reference、hash、duplicate、date、negative value、SVG、private field、GATE-001〜006、tag auditの全件合格を必須とする。

`releaseMode=compliance_purge`はdeletion／retention／GATE invalidation eventだけが起動できる。build開始時のlatest `releaseId`とmanifest hashを`purgeBaseReleaseId`／`purgeBaseManifestSha256`へ固定し、trigger type／eventIdを記録する。publish直前のlatestがbase ID／hashと異なる場合は`stale_purge_base`でrejectし、新しいbaseから再生成する。purgeはbaseと同じ`schemaVersion`／`normalizationVersion`を使い、video集合をsubsetにだけできる。保持するvideoのtitle、sourceKind、metadataStatus、date値／精度、各statusのconditional coverage fields、`provenance.metadataSource`／optional `provenance.sourceUpdatedAt`、およびsearch `titleTokens`はbaseとbyte-equivalentな値を保つ。`provenance.derivedLabel`だけは`diopside_generated`から`none`への単調な安全化変更を必須とし、release envelope／path／hash以外のsource fieldを変更しない。tag／derived fieldとartifactの削除、video行の削除、全件削除だけを許可し、新video、metadata更新、token再正規化、tag／derived生成、scope拡張を禁止する。

GATEが無効でも、上記base固定とmonotonic removal diff、tag／derived禁止field 0件、private field 0件、candidateから対象が除外済みであること、旧path／CDN invalidation planが検証済みであることを満たす場合だけreadyにできる。ready前にactive old pathの不存在を要求しない。publishはlatest pointerをcompare-and-swapで原子的に変更し、その後5分以内に旧public path／CDNをinvalidateしてread-back不存在を検証する。失敗時は削除対象を含まないsafe latestを維持し、旧pathをdenyしてinvalidationをretryする。禁止データを再公開するold releaseへrollbackしない。purgeはproduct acceptance合格に使わない。

normal rollbackはpolicy-compliant release catalogから15分以内に行う。catalogはactiveに加えて直前のpolicy-compliant候補を最低2版、最大90日保持し、候補ごとにreleaseId、manifest hash、publishedAt、policy／deletion validityを記録する。削除・失効で候補が禁止対象を含む場合は保持数より削除を優先し、候補不足をdegradedとしてnormal release acceptanceをfailにする。rollback drillは最新priorが利用不能なfixtureでも2つ目の候補へ切り替える。

HTTP cache contractは、versioned canonical artifactを`Cache-Control: public, max-age=300, immutable`、`/data/latest.json`を`public, max-age=60, must-revalidate`とし、stale-while-revalidateを使わない。ETagは展開後bytesのSHA-256、compressed responseは`Vary: Accept-Encoding`を返す。normal old releaseは最大90日でorigin objectをexpireし、その時点でCDNをinvalidateする。deletion／source失効／Policy失効は90日を待たず、5分以内にdeny／invalidateする。publish後のpointer／artifact不一致とpurge後のold origin／CDN read-back成功を5分以内に0件へする。

### SPEC-OPS-DELETION-001: 削除requestのlifecycle

- 要件: DR-001、DR-005、DR-008、NFR-COMP-004
- 種別: privacy operation
- 対象: target operations
- 確度: confirmed
- 出典: SRC-YT-POLICY
- 受け入れ: AC-SEC-02、AC-DATA-02

deletion state transitionは次を正とする。create受付と全transitionでactor、from／to、reason、correlationId、時刻をaudit eventへ追記し、receivedAt／deadlineを変更しない。各create／transitionは、clientへ受付・遷移成功を応答する前に、requestId、元のreceivedAt／deadline、全6 layer scope、identity resolution status、暗号化identity参照／key参照、layer別status、attempt、auditEventRefを別failure domainのappend-only compliance ledgerへ同期追記する。ledger追記失敗時はtransitionをcommitせずretryable errorにする。

| From                    | Trigger／guard                           | To                                      | `retryDeletionRequest` rule                                |
| ----------------------- | ---------------------------------------- | --------------------------------------- | ---------------------------------------------------------- |
| create                  | server受付、全6 layer初期化              | `received`                              | not applicable                                             |
| `received`              | identity解決不能／解決済み／対象外承認   | `identity_pending`／`ready`／`rejected` | 409                                                        |
| `identity_pending`      | corrected identityを伴うretry            | `received`                              | `correctedEncryptedIdentityReference`必須                  |
| `ready`                 | deletion worker開始                      | `deleting`                              | 409                                                        |
| `deleting`              | 対象layer処理終了／1件以上失敗           | `verifying`／`partial_failure`          | 409                                                        |
| `verifying`             | 全6 layerとCDN不存在／残存または検証失敗 | `completed`／`partial_failure`          | 409                                                        |
| `partial_failure`       | reason付きretry                          | `deleting`                              | failed／unverified layerだけ再実行、corrected identity禁止 |
| `completed`／`rejected` | terminal                                 | 同state                                 | 409 `invalid_state`                                        |

identity_pending retry以外でcorrected identityを受け取った場合と、許可state以外のretryは`409 invalid_state`とし、元recordを変えない。nonterminal request recordはrequestId、receivedAt、deadline、全6 layer scope、対象identityの暗号化参照、identity resolution status、layer別status、attempt、lastErrorCode、actor、auditEventRefを持つ。completed／rejected recordは暗号化参照とdata keyを破棄し、非可逆tombstone hashとterminal audit chainだけを持つ。

- requester確認と対象特定に必要な最小情報だけをprivate領域へ置く。
- ready以降は対象を除くcompliance purge candidateを検証し、latestを原子的に切り替えてから旧public path／CDNをinvalidateし、read-back不存在を確認する。その後processed、control、raw、backup policyの順序と結果をlayer別に記録する。
- partial failureは成功layerを戻さず、未完了layerだけを同じrequestIdでidempotent retryする。
- public／CDN除去後もprivate deletionが未完了ならcompletedにしない。
- 全layerの不存在確認とCDN purge確認後だけcompletedにし、receivedAtから7日以内を受入基準にする。
- rejectは対象外理由とapproverを必要とし、silently dropしない。

### SPEC-OPS-RECOVERY-001: Backup、redrive、災害復旧

- 要件: BR-006、FR-EXP-004、NFR-REL-004、NFR-MAINT-003
- 種別: recovery operation
- 対象: target operations
- 確度: confirmed
- 出典: SRC-UC-REPO
- 受け入れ: AC-OPS-02、AC-NFR-03

raw manifest、checkpoint、public manifestのbackupからRPO 24時間、RTO 4時間を満たす隔離済み非本番restore testを行う。DLQ redriveは同一inputを重複計上せず、rollbackはlast-known-good releaseを維持する。runbookはdiagnose、stop、restore、validate、resume、evidence captureを含む。

deletion requestのcreateと`received`、`identity_pending`、`ready`、`deleting`、`verifying`、`partial_failure`、terminalへの全transition、deletion tombstone、terminal deletion audit、retention expiry、gate invalidation、cryptographic-erasure key ledgerは通常backupより新しく、別failure domainのappend-only compliance ledgerへ保存する。nonterminal entryはrequestId、元のreceivedAt／deadline、全6 layer scope、identity resolution status、暗号化identity参照／key参照、layer別status、attempt、audit／correlation参照を持つ。terminal時はidentity data keyを破棄し、ledger内の暗号文も復号不能にする。

restoreはnetwork／publishを遮断したquarantineへ行い、(1) backup hash／schema検証、(2) backup cutoff後を含むcompliance ledger全件replay、(3)破棄済みkeyを再作成しないことの検証、(4)全storage version／replica／public release／CDNから禁止対象の不存在確認、(5)containment purge／last-known-goodの再構築、(6)gate再評価、の順を完了するまでresume／publishしない。ledgerから復元したnonterminal deletionは同じrequestId、receivedAt、deadline、state、layer別statusで再開し、deadlineを延長せず未完了layerだけを冪等に処理する。request重複、完了layerの再生成、削除対象の再公開を許さない。pre-deletion backup、backup後かつterminal前のin-flight deletion、expired retention data、invalidated gate artifactを含むfixtureで復活0件を証明する。
