## 1. 目的

本文書は、`docs/spec` に置く要求・仕様・受け入れ条件・トレーサビリティの境界と正本順位を定義する。仕様の完成と製品実装の完成は区別する。文書に未実装要件や未解決ゲートが残る場合も、状態、影響、Owner、検証方法を明示し、実装済みまたは合格済みと誤認させない。

## 2. 対象範囲と現在地

### 2.1 製品ターゲット

製品ターゲットは、白雪巴本人の公開YouTubeアーカイブと、実出演・共同制作を精査済みの外部チャンネル公開動画を「探す・つかむ・見返す」ための次の能力を含む。

- モバイルファーストの公開Webフロントエンド
- 動画メタデータ、公開チャット、公開コメントの収集
- 正規化、集計、タグ付け、wordcloud、タイムスタンプ候補の生成
- 字幕・保存コメントからの本文言及と、Wiki・共演者／group member側チャンネルからの外部コラボ発見・review
- version付き静的公開データの生成・検証・切替
- 個人開発者1名で実行できる管理、監視、復旧、規約対応

### 2.2 このリポジトリの現行実装

このリポジトリはReact frontend、YouTube collector、processor、tagging pipeline、atomic static exporter、IAM operator CLI、AWS CDKを同一repoで所有する。canonical normal／compliance purge contract、SQS job、quota／gate制御、監視、削除purgeは実装済みであり、`task verify`とGitHub Actionsで検証する。production公開そのものはGATE-001〜006、外部承認、実AWSのpost-deploy smokeが未完了なので実施済みとは扱わない。

現行フロントは`/data/latest.json`から同一releaseIdの`/data/releases/{releaseId}/...`だけを読み、旧3 pathを互換aliasとして使用しない。契約は [22.system-specifications.md](22.system-specifications.md) で定義する。

### 2.3 タグ資料の位置づけ

`.workspace/tags.zip` は2026-07-11時点の移行資料・分類定義・本人動画／本文言及／外部コラボ発見・付与結果・監査用Excelをまとめた全21ファイルの入力snapshotであり、Git管理された永続正本ではない。規範内容は [30.tagging-requirements.md](30.tagging-requirements.md) へ統合し、各収録物のhash、役割、観測値、不整合は [31.tagging-source-traceability.md](31.tagging-source-traceability.md) で固定する。

## 4. 正本順位

同じ対象について記述が競合する場合は、次の順で解決する。

1. 法令、YouTube API Services Terms／Developer Policies、利用許諾などの外部強制条件
2. 本索引に定義した境界と、[20.requirements-definition.md](20.requirements-definition.md) のシステム要求
3. 対象領域の詳細正本。タグ要求は [30.tagging-requirements.md](30.tagging-requirements.md)、UI token／component／responsive／interaction／accessibilityは `docs/design/design-system.md`
4. [22.system-specifications.md](22.system-specifications.md) の画面flow、data、API、job、運用contract。UI detailはdesign systemを上書きしない
5. [23.acceptance-and-verification.md](23.acceptance-and-verification.md) の合否条件。要求や詳細正本を緩和しない
6. 現行コード・テストが示す実装事実
7. `.workspace` の入力資料と生成snapshot

上位文書が下位の事実と異なる場合、下位を無視して「covered」とは判定しない。[25.gaps-and-decisions.md](25.gaps-and-decisions.md) に差分、影響、解消条件を登録する。

## 5. 要件と仕様の記法

- `BR-*`: 業務要求
- `CON-*`: 制約
- `GATE-*`: 外部承認またはproduction移行ゲート
- `FR-*`: 機能要求
- `DR-*`: データ要求
- `NFR-*`: 非機能要求
- `SPEC-*`: 実装方式に依存しない具体的な画面・データ・API・運用仕様
- `AC-*`: Given／When／Thenで判定する受け入れ条件
- `GAP-*`: 未実装、不整合、証拠不足、外部依存
- `DEC-*`: 競合を解消した決定

各要求は、個別行または文書内のfamily default／trace tableとの結合によりID、Type、Priority、Confidence、Source、Verificationを持つ。各ACもregistry、attribute default、related requirementのSource traceとの結合によりType、Confidence、Source、Related requirements、検証手段を持つ。属性の所在を暗黙にせず、`inferred`な閾値は確定値に混ぜない。

## 6. トレーサビリティ規則

重要な項目は次の両方向で追跡可能にする。

```text
Source -> BR/CON/GATE/FR/DR/NFR -> SPEC -> AC -> command/test/evidence
evidence -> AC -> SPEC -> requirement -> Source
```

範囲表記だけで個別要件の証拠を代用しない。未実装のtest名、存在しないcommand、実施していないmanual reviewは合格証拠に数えない。

## 7. 変更管理

1. 変更理由と影響するIDを特定する。
2. 要求、詳細仕様、AC、trace、gapを同じ変更で同期する。
3. 公開schemaの非互換変更はmajor versionと移行期間を定義する。
4. 外部Policy、production、credential、予算、削除に関わる変更は人の明示承認を得る。
5. Markdown、リンク、ID参照、構造化データ例を検証し、未実施の検証を記録する。

## 8. 文書完成の判定

仕様書群は、次をすべて満たす場合に文書としてcompleteと判定する。

- 依頼された全資料がSource台帳にあり、各項目に反映先または非採用理由がある。
- 全要求が少なくとも1つのSPECと検証経路へ追跡できる。
- 全SPECが要求へ逆追跡できる。
- 正常、異常、境界、権限、復旧、security、privacy、accessibility、performanceの該当ケースがACにある。
- 現行実装、目標契約、入力snapshotを混同していない。
- 未実装・未確定・外部依存をgapとして明示し、完了済みと記載していない。
- 文書検証が成功し、レビュー証跡が事実と一致する。

この判定は製品リリースのDefinition of Doneを満たしたことを意味しない。
